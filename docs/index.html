PART 1Install these four things

In order. Each takes a few minutes. You do not need to understand them yet — install now, understand later.

1

Git — the undo button

Tracks every version of your files, so you can never permanently break anything.

  • Windows: git-scm.com/download/win — accept every default.
  • Mac: open Terminal and run xcode-select --install.

Then open a terminal — Windows: search for Git Bash; Mac: Terminal — and check it worked:

git --version

You should see a version number. If it says “command not found”, close the terminal, open a new one, and try again — a fresh terminal is needed to notice a new install. That is true of every install on this page.

Then tell Git who you are, once, forever:

git config --global user.name "Your Name"
git config --global user.email "your@email.com"
2

Python — runs the build

Turns your template plus your data into the finished page. Download from python.org/downloads.

Windows — the one box everybody misses

Tick “Add Python to PATH” on the very first installer screen. Skipping it causes every python is not recognised error you will hit later. If you missed it, re-run the installer and choose “Modify”.

Mac — it is python3, not python

On a Mac the command is python3. Plain python does not exist and will say “command not found”. Everywhere on this page you see python, type python3 instead. Same for the build command later.

python --version     # Windows
python3 --version    # Mac
3

Node.js — local preview

You need it for exactly one command: a local web server, so you can look at your page before publishing it. Get the LTS version from nodejs.org.

node --version
4

Claude Code — does the work

Needs a Claude account on a paid plan (Pro or Max). Then:

npm install -g @anthropic-ai/claude-code

Start it, and sign in when prompted:

claude
This matters more than it looks

Claude Code only sees the folder you launched it from, and everything underneath. Always cd into your project folder first, then run claude. Type /exit to leave.

And one account: GitHub

Not an install, but you need it. Sign up free at github.com. This is where your code lives and where your finished pages get hosted, at no cost.

PART 2Run it

Type these one line at a time, pressing Enter after each. The first command downloads this kit into a new folder; the rest build it and show it to you.

git clone https://github.com/phuazz/claude-code-starter-kit.git
cd claude-code-starter-kit
python scripts/pipeline.py
npx serve docs
Two things that will happen and are not errors

Where did the folder go? Wherever your terminal was pointing — usually your home folder, so C:\Users\you\claude-code-starter-kit or /Users/you/claude-code-starter-kit. Type pwd at any time to see where you are.

“Ok to proceed? (y)” — the first time you run npx serve, Node offers to fetch the serve package. Press y and Enter. It only asks once.

Open the URL it prints — usually http://localhost:3000 — and click through to the demo. You just built the same page you saw above, on your own machine. Press Ctrl+C in the terminal to stop the server when you are done looking.

Now look at what actually happened, because this is the whole architecture:

FileWhat it is
template.htmlThe source. Small, and the only HTML you ever edit by hand.
data/metrics.jsonThe data. Plain numbers, no code.
scripts/pipeline.pyThe build. Injects the data into the template and writes the result.
docs/demo.htmlThe output. Generated. Never edit it. It can be huge — nobody reads it.

That split is why this stays cheap to maintain. Change the look, edit the template. Change the numbers, edit the JSON. The two never fight, and you can regenerate the output at any time from the other two.

Do this now — it is the whole job

Open data/metrics.json in any text editor, change 48,210 to something silly, save, run python scripts/pipeline.py, and reload the demo. Edit → build → look. That loop is everything. The rest of this page is detail.

PART 3Your first five minutes inside Claude Code

Nobody explains this part, and it is where most people get confused. Start it from inside the kit folder:

cd claude-code-starter-kit
claude

You now have a prompt. Type a request in plain English and press Enter. Try this one — it is real work, and it is safe:

Read README.md, then change the dashboard title from "Sample Dashboard"
to "My Dashboard". Show me the plan before you start.

It will ask permission — this is the important bit

Before Claude Code edits a file or runs a command, it stops and asks you, showing exactly what it wants to do. You approve it, reject it, or tell it to stop asking for that kind of thing in this project. Nothing happens to your files without you saying yes. That is why it is safe to let it work.

Read the diff it shows you. Approving without looking is how people end up surprised.

The five things worth knowing on day one

EscInterrupt it mid-answer. Use this the moment it goes the wrong way — do not wait politely for it to finish.
/clearWipe the conversation and start fresh. Do this between unrelated tasks; a stale conversation makes answers worse.
/exitQuit.
Shift+TabCycles how much it asks permission for — including plan mode, where it can read and explore but cannot change anything. Good for “explain this to me” sessions.
Just talk“That is wrong because X” is a valid message. So is “no, do it the other way”. It is a conversation, not a command line.
If it cannot see your files

You launched it from the wrong folder. Type /exit, cd into the project, and run claude again. This is the single most common beginner problem.

PART 4Publish it free

Your page goes on the internet, at no cost, on GitHub Pages. The page you are reading right now is hosted exactly this way.

  1. Create a new repository on GitHub. Do not tick “Add a README” when it offers.
  2. GitHub then shows you your repository URL. Point your folder at it. You cloned this kit in Part 2, so your folder currently points at mine — this command repoints it at yours:
git remote set-url origin https://github.com/<you>/<repo>.git

Replace <you> and <repo> with your own. Then send your files up:

git add -A
git commit -m "Initial commit"
git push -u origin main
Starting a project from scratch instead?

If the folder is your own and has never been cloned, it has no remote yet, so use git remote add origin <url> — add rather than set-url. If Git ever says remote origin already exists, you want set-url.

  1. On GitHub: Settings → Pages → Source: Deploy from a branch → Branch main, folder /docs → Save.
  2. Wait a minute or two, then visit https://<you>.github.io/<repo>/.

That is it. Every push updates the page. Free, permanent, no server.

The kit also ships a GitHub Action (.github/workflows/build.yml) that reruns the build for you on every push and once a week. Once that works, add a step that fetches fresh data before the build and your page updates itself forever, without you touching it.

PART 5The actual method

Everything above is plumbing. This part is why it keeps working after week one, and it is the part people skip.

The whole idea in one line

The productivity does not come from clever prompting. It comes from four files that mean you never explain yourself twice.

Claude Code reads these automatically at the start of every session, so it begins already knowing your rules instead of guessing at them. Open them — they are short, and reading them is the fastest way to understand this whole page:

CLAUDE.md Open file →

Your standing rules. Read automatically, every session.

PROMPT_TEMPLATE.md Open file →

How to ask for work. Includes what to leave out, and a before/after.

KICKOFF_TEMPLATE.md Open file →

How to start something new — the interview trick.

design.md Open file →

How it should look. Colours, fonts, layout, chart defaults.

CLAUDE.md — your standing rules

Read automatically at the start of every session in that folder or any folder inside it. This is where you write down what you would otherwise repeat in every conversation: how you want writing to sound, what your build command is, what Claude must never do.

The single highest-value line in mine is a guardrail: never open generated output larger than 500KB. Generated files are enormous, and one accidental read wastes the whole session.

Write your guardrails as rules, not as reminders you have to remember to give. That is the difference between a setup that scales and one that does not.

Excerpt — the guardrails section 
### Hard rules

- **NEVER open generated output larger than 500KB.** It wastes the
  session. Work on `template.html` instead.
- **Check file size before opening anything unfamiliar.** Over 200KB,
  stop and ask me to point you at the source file.
- **Pattern consistency.** When fixing a pattern, apply the fix to every
  instance of that pattern, and audit for others before calling it done.
- **Never commit credentials, API keys, or personal identifiers.**
How to grow it — do not write it up front

Start nearly empty. Add a rule the moment you catch yourself correcting Claude twice on the same thing. That is the only trigger you need. Mine reached a couple of hundred lines over months without me ever sitting down to write it.

PROMPT_TEMPLATE.md — how to ask for work

A good prompt looks like a well-scoped engineering ticket, not a spell. Context, constraints, success criteria, output format. Everything else is decoration. Use this for most things:

Project: <name>. Stage: <bug fix | new feature | tweak>.
Change: <what should be different>.
Must: <the non-negotiables>.
Show me the plan before you start.

That last line does more than all the rest combined. It costs one turn and catches a misunderstanding while it is still cheap to fix.

What to leave out entirely — all decoration, none of it helps:

  • Personas — “you are a world-class engineer”
  • Stakes and incentives — “this is critical”, “I will tip you $200”
  • Emotional framing — “take a deep breath”
  • Vague adjectives with no checklist — “rigorous”, “thorough”

The test: if a phrase does not change what a correct output would look like, it is decoration. Replace adjectives with checklists — “rigorous review” becomes the three specific things rigour means here.

Excerpt — the same request, before and after 
BEFORE — theatre

Act as a world-class engineer. This is critical work. Take a deep
breath and rigorously review my data pipeline for robustness. I bet
you cannot find real issues.

AFTER — actual instruction

Review the data pipeline. Check specifically:
(1) what happens when the input JSON is missing a field,
(2) whether dates are parsed with a library rather than string slicing,
(3) month and year boundary handling,
(4) what happens on an empty input file,
(5) whether the build fails loudly or writes a broken page.
Flag anything uncertain rather than papering over it.

Same length. Zero persona. Dramatically better output, because
every clause constrains what "done" looks like.

KICKOFF_TEMPLATE.md — how to start something new

The best trick in the kit. When you have a goal but not a spec, do not describe it badly and hope. Make Claude interview you. You end up with a better-specified project than you would have written alone, because answering a pointed question is far easier than imagining the whole thing up front.

We are starting a new <build | tool | dashboard> at <path>.
I have the goal but not a full spec. Before proposing any code or plan,
interview me to scope it — ONE question at a time, waiting for my answer
before the next.

Cover, in roughly this order, and stop asking once an item is settled:
- Objective: the one decision or output this must produce.
- Users: who looks at this, and what they do differently as a result.
- Data: what goes in, where from, what shape, how often it changes.
- Success criteria: must-haves, nice-to-haves, out of scope.
- Constraints: anything that must not break.
- Failure modes: the three ways this could be silently wrong, and the
  guard for each.

Ask only what you cannot reasonably infer. Where you can infer, state the
assumption and move on. Do not write code during the interview. When the
spec is settled, output the completed prompt template and a multi-turn
plan, then stop and wait for my go-ahead. Push back if any answer looks
wrong. Read README.md and CLAUDE.md in this folder first.

Two clauses make this work, and both are easy to drop. “ONE question at a time” — without it you get twenty questions in a wall of text and answer none of them properly. “Do not write code… then stop and wait” — without it, enthusiasm takes over and you are reviewing an implementation of the wrong thing.

design.md — how it should look

One file with your colours, fonts, and layout rules. Reference it in a prompt — “style per design.md” — and everything you build looks like it came from the same place, without you art-directing anything. Copy the token block into each project rather than inventing new values.

One rule keeps it alive: if you need a value the system does not have, add it there first, then use it. The moment one project invents its own blue, the system is dead.

PART 6Habits that matter more than they look

Let it work, then check it. Claude Code writes the file, runs the build, and reads the error. Your job is to check the output and push back — not to supervise each keystroke. Ask for a plan on anything non-trivial; let the small stuff run.

Commit constantly. git commit after every working change. It is your undo button. Nothing you do is scary if the last good version is one command away.

Every project gets a README. What it is, how to build it, where the data comes from, what is broken. This is what you read when you come back in three months having forgotten everything — and it is the first thing Claude reads too. One file, two audiences.

Push back. When the output looks wrong, say so plainly. “That is wrong because X” gets a fix; “hmm, not quite” gets a shuffle.

Iterate across turns. Do not try to one-shot the perfect prompt. Nobody writes it first time, and trying to is slower than correcting a draft.

PART 7When it goes wrong

Every one of these has happened to everybody. None of them mean you have broken anything.

What you seeWhat to do
python is not recognised Windows: you missed “Add Python to PATH”. Re-run the installer, choose “Modify”, tick the box. Mac: type python3 instead of python.
command not found
(git, node, claude)		 Close the terminal and open a new one. A terminal only notices new installs when it starts.
remote origin already exists You cloned this kit, so origin points at mine. Use git remote set-url origin <your url> instead of git remote add.
Ok to proceed? (y) Not an error. Node is fetching the serve package. Press y and Enter.
Your Pages URL 404s Give it two minutes — the first build is slow. Then check Settings → Pages really says branch main and folder /docs.
Claude cannot see my files You launched it from the wrong folder. /exit, cd into the project, run claude again.
Claude changed the wrong thing git checkout -- <filename> throws away the change. This is why you commit often — and why you read the diff before approving.
The page is blank Right-click → Inspect → Console. Whatever red text is there, paste it to Claude Code and ask it to fix it. That is a completely normal way to work.
